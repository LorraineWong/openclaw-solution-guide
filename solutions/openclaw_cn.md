## 0. 核心价值

> OpenClaw = 让AI从“对话工具”升级为“执行系统”

其核心能力可以抽象为：

> 理解 → 决策 → 执行

即：

- 理解用户意图  
- 决策执行路径  
- 调用系统完成操作  

该能力贯穿系统架构、执行链路与应用场景，是OpenClaw的核心价值。

---

## 1. 背景

在传统企业信息化体系中，系统主要用于记录与处理数据，而实际业务执行仍然依赖人工完成。例如，客服需要手动查询订单，员工需要人工提交流程，管理层依赖数据团队提供分析结果。这种模式本质上是“系统存数据，人来执行”。

随着大模型技术的发展，企业已经具备通过自然语言完成复杂任务的能力。但在实际应用中，大多数AI仍停留在“对话工具”阶段，仅用于回答问题，而无法真正参与业务执行。这使得AI的价值更多停留在辅助层，而难以进入核心业务流程。

因此可以归纳出一个关键问题：

> **AI具备理解能力，但不具备执行能力**

---

### 1.1 方案概述

为了解决上述问题，OpenClaw 提供了一种将“语言理解”转化为“实际执行”的系统化方案。

以“查询订单状态”为例，不同方式的执行差异如下：

| 模式 | 执行方式 | 特点 |
|------|----------|------|
| 传统方式 | 用户 → 客服 → 登录系统 → 查询 → 回复 | 完全依赖人工操作 |
| AI对话工具 | 用户 → AI → 生成回答 | 无法访问真实数据，结果不准确 |
| OpenClaw | 用户 → Agent → 调用订单系统API → 获取数据 → 返回结果 | 自动执行并返回真实数据 |

从上述对比可以看出，问题的关键不在于AI是否“更聪明”，而在于是否具备“执行能力”。OpenClaw 通过将模型能力与企业系统连接，使自然语言可以直接触发实际操作，从而使AI能够参与业务流程，而不仅仅是提供建议。

---

### 1.3 方案定位

从整体AI体系来看，不同能力可以理解为逐层递进的结构：底层模型提供理解与生成能力，其上通过框架构建逻辑，通过应用平台提供交互能力，而这些能力大多停留在“理解与响应”的层面。

OpenClaw 所处的位置，是在此基础上进一步向“执行”延伸。它通过将模型能力与企业系统连接，使AI能够直接调用业务系统、获取数据并完成操作，从而将AI从“对话工具”升级为“执行系统”。

因此，从定位上来看，OpenClaw 并不是单一模型或应用，而是：

> **一个面向企业业务的AI执行系统，用于将理解能力转化为实际执行能力**

---

## 2. 系统架构

在理解 OpenClaw 的系统架构时，关键不在于具体组件的划分，而在于系统如何将“自然语言请求”转化为“实际业务执行”。一个完整请求的执行路径如下：

用户  
↓  
Gateway（请求入口）  
↓  
Agent（任务理解与决策）  
↓  
Model（语言理解与推理）  
↓  
Tools / Skills（调用能力）  
↓  
企业系统（CRM / ERP / 数据平台）  
↓  
返回结果  

与传统系统不同，OpenClaw 并不是简单地接收输入并返回结果，而是通过一条完整的执行链路，将用户的语言逐步转化为可执行的操作。这一过程本质上是从“理解问题”走向“完成任务”的转化过程。

---

### 2.1 架构整体理解

从整体上看，OpenClaw 可以理解为一个围绕“执行”构建的分层系统。不同层级之间并非孤立存在，而是围绕同一个目标协同工作，即将用户意图转化为实际操作。

系统首先通过统一入口接收请求，并将其引入执行流程；随后由执行层对请求进行分析与判断，在需要理解或生成内容时调用模型能力，在需要完成具体操作时通过工具与企业系统交互，最终返回执行结果。

为了更清晰地理解这一结构，可以从功能上进行简化划分：

| 层级 | 作用 |
|------|------|
| 接入层 | 接收请求，作为系统入口 |
| 执行层 | 负责任务理解与执行决策 |
| 能力层 | 提供语言理解与推理能力 |
| 工具层 | 连接企业系统并完成操作 |

这种分层方式的关键在于，将“理解能力”与“执行能力”进行分离，并通过统一的执行逻辑进行连接，使系统既具备灵活性，又具备可控性。

---

### 2.2 执行链路

当用户发起一个请求时，系统会按照固定的逻辑路径运行。

请求首先进入系统入口，随后由执行层对其进行分析与判断。在这一过程中，如果涉及语义理解或内容生成，系统会调用模型能力；如果涉及实际操作，则通过工具调用企业系统接口，获取数据或执行动作。最终，系统将结果整理并返回给用户。

这一过程可以概括为：

> 请求 → 理解 → 决策 → 执行 → 返回

从本质上看，这条链路完成的是一次从自然语言到结构化操作的转换，使用户可以通过语言直接驱动系统行为。

---

### 2.3 架构核心特点

与传统对话系统相比，OpenClaw 的架构体现出明显差异。

首先，系统不再以模型为中心，而是以“执行”为核心。模型仅提供理解能力，而真正的控制与调度发生在执行过程中。

其次，通过与企业系统的连接，AI不再局限于生成回答，而是具备了调用外部能力的接口，从而能够参与实际业务流程。

此外，通过分层设计，系统实现了能力解耦，使模型、执行逻辑以及系统接入可以独立演进，从而提升整体的灵活性与扩展能力。

---

### 2.4 架构映射（企业视角）

为了便于企业理解系统如何落地，可以将 OpenClaw 架构映射到实际IT体系中：

| OpenClaw 组件 | 企业系统映射 |
|---------------|-------------|
| Gateway | API网关 / 统一接入层 |
| Agent | 业务编排层 / AI中台 |
| Model | 大模型服务（云或本地） |
| Tools / Skills | 微服务接口 / API服务 |
| 企业系统 | CRM / ERP / 数据平台 |

在企业IT架构中，OpenClaw 更接近于一个“AI中台”，负责连接用户请求与业务系统能力。

## 3. 部署与使用

在企业环境中，系统是否能够快速部署并投入使用，是决定方案可行性的关键。因此，本章节从环境准备、部署启动、基础配置以及快速使用四个方面，说明系统从“启动”到“可用”的完整路径。

---

### 3.1 环境要求

OpenClaw 支持在个人电脑、服务器或一体机环境中运行。不同环境在资源规模上有所差异，但整体部署方式一致。

在基础环境上，需要具备以下条件：

- 已安装 Node.js（建议 LTS 版本）  
- 具备网络访问能力（用于模型调用或依赖下载）  
- 服务器环境需开放访问端口（默认 18789）  

对于一体机场景，上述环境通常已预集成，用户可以直接进入使用阶段。

---

### 3.2 部署与启动

完成环境准备后，可以通过以下步骤完成系统部署与启动：

安装 OpenClaw：

```bash
npm install -g openclaw
```

初始化系统：

```bash
openclaw onboard --install-daemon
```

启动服务：

```bash
openclaw gateway --port 18789
```

启动完成后，通过浏览器访问：

```bash
http://localhost:18789
```

如果为服务器部署，则使用对应服务器 IP 地址访问。

---

### 3.3 初始化与配置

系统启动后，需要完成基础配置，使系统具备可用能力。

首先需要配置模型，这是系统能够运行的前提。用户可以选择接入云端模型（如 OpenAI、Claude），或配置本地模型（如 Qwen、DeepSeek）。模型配置完成后，系统即可具备语言理解与生成能力。

随后，可以在系统中创建一个 Agent，用于定义系统的基本角色，例如“通用助手”或“客服助手”。Agent 是系统执行能力的入口，后续所有操作都将围绕 Agent 进行。

---

### 3.4 快速使用（Quick Start）

完成基础配置后，可以通过一个简单流程快速验证系统是否可用：

1. 打开系统界面  
2. 确认模型已配置成功  
3. 创建一个 Agent  
4. 输入简单问题进行测试，例如：  
   - 你好  
   - 帮我总结一下人工智能  

如果系统能够正常返回结果，则说明系统已经成功运行。

---

### 3.5 使用路径

在系统能够正常对话后，可以逐步扩展其能力，使其进入实际业务场景。

首先通过对话验证系统基础能力；随后接入企业数据或业务系统，使系统具备获取真实数据的能力；在此基础上，通过工具配置，使系统能够执行具体操作，例如查询数据或触发流程。

这一过程可以理解为一个逐步演进的路径：

> 从“系统可用”，到“能力扩展”，再到“业务落地”

系统的价值也正是在这一过程中逐步体现出来。

## 4. 核心机制（Agent 与执行能力）

在前文中已经说明，传统AI主要停留在“对话”层面，而无法真正参与业务执行。本章节关注的核心问题是：OpenClaw 为什么能够实现从“理解问题”到“完成任务”的能力跃迁。

从本质上看，这一能力并不是来源于模型本身，而是来源于系统对“理解、决策与执行”三类能力的重新组织。

---

### 4.1 从理解到执行的差异

在仅使用大模型时，系统的能力主要体现在语言理解与内容生成上。模型可以根据输入生成合理的回答，但其输出本质上仍然是“文本”，而不是“动作”。

这意味着，即使模型能够理解用户意图，也无法直接访问企业系统或执行具体操作。因此，传统AI更适合用于辅助决策，而难以参与实际业务流程。

OpenClaw 所解决的，正是将这一“理解能力”进一步延伸为“执行能力”的问题。

---

### 4.2 执行能力的来源

要实现从理解到执行的转变，需要在模型之外引入新的机制。

在 OpenClaw 中，用户请求首先被理解，然后由系统判断是否需要执行操作；当需要执行时，系统会调用对应的外部能力（如接口或数据服务），并将执行结果返回给用户。

这一过程可以理解为三类能力的协同：

| 能力 | 作用 |
|------|------|
| 理解能力 | 识别用户意图 |
| 决策能力 | 判断是否需要执行以及如何执行 |
| 执行能力 | 调用系统完成具体操作 |

通过这种方式，系统不再停留在“生成答案”，而是能够完成实际动作。

---

### 4.3 从对话到执行的转变

在引入执行机制后，系统能力发生了本质变化。

原本，用户与AI之间的交互仅限于问答；而在执行机制下，用户可以通过自然语言直接触发系统行为，例如查询数据、调用接口或执行流程。

这一变化的关键在于：

> **用户表达的是“意图”，系统完成的是“操作”**

因此，交互方式虽然仍然是对话，但其结果已经从“信息反馈”转变为“任务完成”。

---

## 5. 系统集成

在系统运行之后，是否能够接入企业现有系统，决定了AI能否真正产生业务价值。

OpenClaw 的核心目标，是将企业系统中的数据与能力接入，使AI能够访问数据并执行实际操作，而不仅仅停留在对话层面。

---

### 5.1 集成对象

在企业环境中，常见需要接入的系统包括业务系统、数据系统以及内部服务。这些系统承载着企业核心数据与业务能力，是AI执行的基础。

其中，业务系统如 CRM、ERP 或订单系统，提供具体业务操作能力；数据系统如数据库或数据平台，提供结构化数据支持；内部服务则通常以 API 或微服务形式存在，用于承载企业内部逻辑。

---

### 5.2 集成方式

系统通过接口或数据连接完成集成，其本质是将外部能力转化为系统可调用的资源。

| 集成方式 | 说明 |
|----------|------|
| API 接口 | 调用业务系统能力（查询、创建、更新等） |
| 数据访问 | 查询或写入数据库数据 |
| 服务调用 | 调用内部服务或微服务能力 |

---

### 5.3 集成过程

系统集成通常围绕具体业务场景展开。首先明确需要实现的业务能力，其次梳理企业系统中对应的接口或数据来源，然后完成接入并进行验证。

这一过程的关键，在于将原本分散在各个系统中的能力，整理为清晰、可调用的接口，使AI能够基于用户意图正确触发。

---

### 5.4 集成示例（以订单查询为例）

为了更直观地理解系统集成方式，可以以“查询订单状态”为例说明。

在企业系统中，通常已经存在订单查询接口，例如：

GET /api/order/{id}

在完成系统接入后，该接口可以作为系统可调用能力的一部分。

当用户输入“帮我查询订单123状态”时，系统会识别该请求为订单查询操作，并调用对应接口获取数据，随后将结果整理并返回给用户。

在这一过程中，用户只需通过自然语言表达需求，而系统负责完成接口调用与结果处理，从而实现自动化查询。

---

### 5.5 Skill 定义示例

在 OpenClaw 中，企业系统能力需要封装为 Skill，供 Agent 调用。

以订单查询为例：

Skill 名称：OrderQuerySkill  
接口地址：GET /api/order/{id}

输入参数：
- order_id（订单ID）

输出结果：
- status（订单状态）
- amount（金額）
- create_time（创建时间）

示例返回：

{
  "order_id": "123",
  "status": "已发货",
  "amount": 299,
  "create_time": "2026-03-01"
}

通过标准化 Skill 定义，系统可以稳定调用企业能力，实现可复用与可扩展。从本质上看，Skill 是对企业API能力的标准化封装，使AI能够以稳定、可控的方式调用业务系统，从而实现从“理解”到“执行”的闭环。

## 6. 应用场景

在企业环境中，AI的价值不在于“是否智能”，而在于是否能够参与实际业务流程，并完成从“理解问题”到“执行操作”的闭环。

OpenClaw 通过 Agent 调用 Skills，将用户意图转化为系统操作，并在“分析—决策—确认—执行”的机制下完成任务，确保执行过程可控、安全。

---

### 6.1 医疗行业 — 检查预约与患者服务协同

**1. 业务背景**  
在医疗场景中，患者预约与改期通常需要人工处理，涉及查询排班、确认时间及执行修改等多个步骤，同时必须符合权限与合规要求。

**2. 用户表达**  
“我明天是不是有一个检查？如果是上午的话我可能来不了，能不能帮我改到下午或者晚一点的时间？”

**3. 系统处理过程**

| 阶段 | 系统动作 |
|------|----------|
| 理解 | Agent 识别为“检查查询 + 改期请求” |
| 数据获取 | 调用 Patient Info Skill 与 Appointment Query Skill 获取预约信息 |
| 分析 | 调用 Availability Query Skill 查询可用时间段 |
| 决策 | 系统生成可调整方案 |
| 确认 | 用户选择时间，并触发医护审核流程 |
| 执行 | 调用 Appointment Reschedule Skill 完成修改 |
| 返回 | 输出最终结果 |

**4. 系统交互（关键）**  

系统返回：

“您当前预约为明天上午10点，可改期时间为：  
- 下午2点  
- 下午3点  
- 下午4点  
请确认您希望调整的时间。”

用户确认后：

“已为您提交改期申请，医护人员确认后将完成调整。”

**5. 执行结果**  
“您的预约已调整至下午3点。”

**6. 业务价值**  
将原本依赖人工的预约处理流程进行系统化协同，在提升服务效率的同时，确保操作符合医疗权限与合规要求。

---

### 6.2 金融行业 — 交易处理与风控协同

**1. 业务背景**  
金融交易处理涉及查询、风控判断及操作执行等多个环节，传统方式依赖人工排查与处理，效率较低且流程复杂。

**2. 用户表达**  
“我刚刚有一笔转账好像没成功，你帮我看看是怎么回事，如果不是被限制的话能不能帮我再试一次？”

**3. 系统处理过程**

| 阶段 | 系统动作 |
|------|----------|
| 理解 | Agent 识别为“交易查询 + 条件处理请求” |
| 数据获取 | 调用 Transaction Query Skill 获取交易记录 |
| 分析 | 调用 Risk Analysis Skill 分析失败原因 |
| 决策 | 判断是否允许重新执行（风控校验） |
| 确认 | 向用户确认是否执行重试 |
| 执行 | 调用 Transaction Retry Skill 或提交处理请求 |
| 返回 | 输出处理结果 |

**4. 系统交互（关键）**  

系统返回：

“该交易因网络超时失败，可为您重新提交，是否继续？”

用户确认后：

“可以，帮我重新提交。”

**5. 执行结果**  
“交易已重新提交，目前处理中。”

**6. 业务价值**  
将“查询—分析—判断—执行”的流程自动化，在确保风控合规的前提下，大幅提升处理效率与客户体验。

### 6.3 互联网行业 — 数据分析与运营策略执行

**1. 业务背景**  
在互联网业务中，数据分析与策略调整通常需要依赖多个工具与人工判断，难以形成快速闭环。

**2. 用户表达**  
“最近这几天感觉转化率有点掉了，你帮我看看是不是哪个渠道的问题，如果确实表现不好就先帮我停掉”

---

**3. 系统处理过程**

| 阶段 | 系统动作 |
|------|----------|
| 理解 | Agent 识别为“数据分析 + 策略执行请求” |
| 数据获取 | 调用 Data Query Skill 获取关键指标 |
| 分析 | 调用 Analysis Skill 分析趋势变化 |
| 决策 | 调用 Channel Performance Skill 识别低效渠道 |
| 确认 | 向用户提供优化建议并确认执行 |
| 执行 | 调用 Campaign Control Skill 执行策略调整 |
| 返回 | 输出执行结果 |

**4. 系统交互（关键）**  

系统返回：

“转化率下降主要由渠道A导致，建议暂停该渠道投放，是否执行？”

用户确认后：

“可以，先停掉。”

**5. 执行结果**  
“渠道A已暂停投放。”

**6. 业务价值**  
实现从“数据分析”到“策略执行”的闭环，大幅提升运营效率与决策响应速度。

通过上述场景可以看出：

> **AI的核心价值在于参与业务闭环，而不仅是提供信息**

OpenClaw 通过 Agent 与 Skills 的协同，使系统能够完成从理解、分析、决策到执行的全过程，并在权限与确认机制下实现可控执行。

这使AI从对话工具，转变为企业业务流程中的执行能力。

## 7. 安全与运维

在企业环境中，系统是否安全、可控以及可持续运行，是其能否落地的关键前提。

OpenClaw 在设计上以“可控执行”为核心，通过执行控制、权限体系与审计机制的结合，使系统在提升效率的同时，满足企业对安全与合规的要求。

---

### 7.1 执行控制机制

系统采用“分析—决策—确认—执行”的闭环机制，确保所有关键操作在可控范围内完成。

对于涉及数据修改、流程触发或业务执行的请求，系统不会直接执行，而是先生成处理建议，并在获得用户或授权人员确认后再执行操作。

同时，系统支持多种执行模式，以适配不同业务场景：

| 执行模式 | 说明 |
|----------|------|
| 推荐模式 | 仅提供分析与建议，不执行操作 |
| 确认执行 | 经用户或授权人员确认后执行 |
| 自动执行 | 在低风险或已授权场景下自动执行 |

通过执行控制机制，系统在保证效率的同时，有效降低误操作风险。

---

### 7.2 权限与访问控制

系统执行能力严格遵循企业现有的权限体系，不会绕过既有安全机制。

不同角色具备不同操作能力：普通用户仅能进行查询与请求，具备权限的业务人员或系统角色才可执行具体操作。在关键业务场景中，还可结合审批流程，实现“请求—审核—执行”的闭环。

在系统访问层面，所有接口调用均基于企业已有的认证与授权机制进行统一管理，从而确保数据访问与操作行为的安全性。

---

### 7.3 审计与可追溯性

系统对关键操作提供完整的记录与审计能力，使所有执行过程可追溯、可复盘。

从用户请求、Agent决策过程、Skill调用到最终执行结果，均可进行记录与追踪。这不仅有助于问题排查，也满足企业在合规与审计方面的要求。

在出现异常或执行偏差时，可以快速定位问题来源并进行修正。

---

### 7.4 运维与持续优化

系统在运行过程中支持持续监控与优化，以保障稳定性与长期效果。

通过对请求处理、执行成功率及系统性能的监控，企业可以及时发现问题并进行调整。同时，随着业务需求变化，可逐步扩展新的 Skills 与应用场景，实现从试点到规模化应用的演进。

---

### 7.5 本章总结

从整体来看，系统通过执行控制、权限管理与审计机制，构建了一套可控、可追溯的运行体系。

> **在保障安全与合规的前提下，实现AI驱动的业务执行能力**

这使系统不仅“能够使用”，更“值得信任并可长期运行”。
